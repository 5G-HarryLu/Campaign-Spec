CREATE TABLE outbox_msg ( id BIGINT -- Snowflake ID，全域唯一，作為 dedup 錨點 topic VARCHAR(128) -- 'msg-center.campaign-notify' partition_key VARCHAR(128) -- 'member:<member_id>' source_table VARCHAR(128) -- 'free_round_bonus_ticket' operation_type VARCHAR(16) -- 'INSERT' before_data JSON -- INSERT 時與 after_data 相同 after_data JSON -- 完整訊息 payload created_at DATETIME(3) -- UTC_TIMESTAMP() );
多 Pod Relay — SELECT FOR UPDATE SKIP LOCKED
同事確認：文件 Section 9.8 已有此設計，多個 Relay Pod 可以安全並行，不需要 leader election。
-- Relay 的 poll query（支援多 Pod 安全並行） SELECT * FROM outbox_msg WHERE id > :checkpoint ORDER BY id LIMIT 100 FOR UPDATE SKIP LOCKED ← 每個 Pod 搶不同的 row，互不衝突 -- 成功 produce 後 DELETE，失敗保留等下次 -- delete-after-publish 本身保證不遺漏 -- idempotency check 保證不重複
id 是整條 pipeline 的 dedup 錨點
outbox_msg.id （Snowflake） ↓ Kafka message key → 防止重複 produce ↓ Valkey SET NX EX 24h → Consumer 快速 dedup ↓ member_inbox.source_ref → DB 唯一約束（最強保證）

table schema
CREATE TABLE member_inbox ( id BIGINT NOT NULL operator_id VARCHAR(36) NOT NULL member_id VARCHAR(36) NOT NULL event_type TINYINT NOT NULL -- 1=FRB, 2=win, 4=system, 8=promo title VARCHAR(255) NOT NULL body JSON NOT NULL status TINYINT DEFAULT 0 -- 0=unread, 1=read, 2=archived source_ref BIGINT NULL, expired_at TIMESTAMP NULL, created_at DATETIME(3) NOT NULL, read_at TIMESTAMP NULL, PRIMARY KEY (id, member_id), UNIQUE KEY uk_source_ref (source_ref), -- 強保證冪等 KEY idx_member_status (member_id, status), -- count query 用 KEY idx_member_created (member_id, created_at), -- 列表排序用 KEY idx_expired (expired_at) -- cleanup job 用 ) PARTITION BY KEY(member_id) PARTITIONS 64;
斷線重連時 member_inbox 的角色
Client 斷線期間，Outbox Worker 繼續正常寫入 member_inbox，完全不受 SSE 連線狀態影響。重連後只需補一次 polling，所有訊息都在 DB 裡等著。
-- 斷線期間 Worker 持續 INSERT member_inbox (status=0) Client 的 SSE 斷了，但 DB 繼續累積 -- 重連後 Step 1: 重建 SSE 連線 Step 2: GET /inbox/count → 回傳 N（斷線期間累積的未讀數） Step 3: GET /inbox?cursor=last_id → 拿回所有未讀訊息 ↑ cursor-based 分頁，訊息量大時不怕

07 / Service Layer
交付到 Game Client
MC API 是唯一入口。SSE 負責即時通知，REST 負責實際資料存取與互動。兩者缺一不可。
Polling 的完整定位（同事補充）
同事指出：Polling 不只是 SSE 的備援，它還有一個更基礎的職責——Game Client 第一次連線時拿到所有歷史訊息。SSE 只能推「連上線之後」的增量，連線前累積的訊息只有 polling 能拿到。
職責
SSE
連線後的即時增量推送（新訊息信號）
Polling
① 初始載入所有歷史訊息
② SSE 斷線後補漏
③ 備援安全網（30 秒）
交付流程圖
REST API 端點 + Cursor 分頁（同事補充）
GET /api/v1/inbox/count?member_id={mid}&status=0 → 未讀數（badge） GET /api/v1/inbox?member_id={mid}&status=0&cursor={id}&limit=20 → 訊息列表（cursor-based 分頁，訊息量大時穩定） → cursor = member_inbox.id，首次不帶 cursor 拿最新 → 回傳 next_cursor 供下頁使用 PUT /api/v1/inbox/{id}/read → 標記已讀 PUT /api/v1/inbox/read-all → 全部已讀 DEL /api/v1/inbox/{id} → 刪除
Cursor-based 分頁比 offset 分頁更穩定：offset 在資料新增時會有「跳過」或「重複」的問題，cursor 用 member_inbox.id 做定位，不受新資料影響。

10 / Review & Recommendations
Review & 建議
以 24/7 系統穩定性與最小元素設計角度的全面評估，包含同事 Alex_Ran-RDDV 的回饋更正。
優點
✓
可靠性設計扎實
Transactional Outbox 確保訊息不丟，每個環節都有持久化或重試機制，沒有 fire-and-forget 的地方。
✓
真正的解耦
Producer 只知道 DB，Consumer 只知道 Kafka，MC API 只知道 DB 和 Valkey。任何一段改動不影響其他段。
✓
Fan-out 架構預留得好
加新 Consumer Group 零成本，不需要改 Producer，未來擴展路徑清晰。
✓
Client 直連 MC API
從 4 層鏈路壓成 1 跳，可靠性提升最明顯的一個決策。
需要改善的點（含同事回饋更正）
①
Relay SPOF — 同事更正：不是 SPOF
原本擔心單一 Pod 掛掉會造成 outbox 累積。
同事確認（Section 9.8）：Relay 使用 SELECT ... FOR UPDATE SKIP LOCKED，多個 Pod 可安全並行，每個 Pod 搶不同的 row，不需要 leader election，不是 SPOF。
②
Valkey Checkpoint 脆弱 — 同事更正：可直接移除
原本擔心 Valkey 重啟後 checkpoint 消失，Relay 可能重複 produce。
同事確認：delete-after-publish 本身已解決問題——已成功 produce 的 row 已刪掉，Relay 重啟後從 outbox_msg 剩餘 row 繼續，自然不重複不遺漏。Checkpoint key 可以直接移除，減少一個 Valkey 依賴。
③
Application-level DLQ 複雜度
Kafka DLQ 需自實作 retry counter（Valkey）+ publish to DLQ topic + 精確控制 commit 時機。任何實作細節出錯輕則重複、重則 pipeline 卡住。採用下方 Outbox Worker 簡化方案後可消除。
④
SSE 連線管理
Pod 重啟或 scale down 時所有 SSE 連線直接斷掉。需 Client 退避重連 + Sticky Session 配合，詳見建議 ⑤⑥。
建議 ③ — Phase 1 合併成單一 Outbox Worker
目前架構（7 個環節）
outbox_msg → Relay → Kafka → Consumer Group → member_inbox
Phase 1 建議（合併）
outbox_msg → Outbox Worker → member_inbox + PUBLISH Valkey
何時引入 Kafka？當出現第二個真實的 Consumer Group 需求（audit、push notification）時再接上，架構天然支援，不需要改 Producer。
Polling 完整定位補充（同事回饋）
Polling 不只是 SSE 備援，它還有一個更基礎的職責：Game Client 第一次連線時取得所有歷史訊息。SSE 只能推「連上線之後」的增量，連線前累積的訊息只有 polling 能拿到。
職責
SSE
連線後的即時增量推送（新訊息信號）
Polling
① 初始載入所有歷史訊息（最基礎職責）
② SSE 斷線後補漏
③ 備援安全網（30 秒）
Cursor-based 分頁補充（同事回饋）
利用 member_inbox.id 做 cursor 分頁，訊息量大時也能穩定運作。Offset 分頁在資料持續新增時會有跳過或重複的問題，cursor 不受影響。
-- 首次請求（不帶 cursor，拿最新 20 筆） GET /inbox?member_id={mid}&status=0&limit=20 -- 翻頁（帶 cursor = 上次最後一筆的 id） GET /inbox?member_id={mid}&status=0&cursor={id}&limit=20 -- 對應 SQL SELECT * FROM member_inbox WHERE member_id=? AND status=0 AND id < :cursor ORDER BY id DESC LIMIT 20

多 Pod 時若沒有 Sticky Session，Client 重連可能換 Pod，Valkey SUBSCRIBE 數量 = Pod 數 × Client 數，PUBLISH 時所有 Pod 都收到但只有一個有 SSE 連線可推，造成浪費。
